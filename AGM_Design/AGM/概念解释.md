## 一、Git 进阶概念

### Hunk（代码块 / 差异块）

`git diff` 输出里每一段以 `@@ -12,5 +12,6 @@` 开头的就是一个 hunk——**一次改动的最小自然单位**，包含上下文行 + 实际增删行。

一个文件如果只改了两处相隔很远的地方，会产生 2 个 hunk。GitButler 的"hunk 级归属"就是说**同一个文件里的不同 hunk 可以属于不同分支**——比如 hunk A 属于"修 bug"分支，hunk B 属于"加新功能"分支。

β 方案的归属引擎就是在做这件事：捕捉文件改动后，切分成 hunk，每个 hunk 归属到发起它的 agent。

### Refs（引用）

分支和 tag 在 Git 里本质都是"指向某个 commit 的指针"，这些指针统称 **ref**，存放在 `.git/refs/` 下。

- `refs/heads/main` → 本地 main 分支
- `refs/remotes/origin/main` → 远端 main 分支在本地的镜像
- `refs/tags/v1.0` → tag v1.0

β 方案用**自定义 ref 命名空间**存虚拟分支：
- `refs/agm/vbranch/<agent_id>` → 每个 agent 一个 vbranch ref
- `refs/agm/metadata/v1` → 元数据的 orphan branch

好处：这些 ref 不会出现在 `git branch` 的默认输出里（除非用 `git branch -a --list 'refs/agm/*'`），不污染用户视觉。

### Plumbing vs Porcelain（底层命令 vs 顶层命令）

Git 命令分两类：
- **Porcelain**（上层/陶瓷）：`git add`、`git commit`、`git merge`——给人用的
- **Plumbing**（底层/管道）：`git hash-object`、`git update-ref`、`git write-tree`——给脚本和工具用的

go-git 暴露的是 plumbing API，所以可以做"不经过正常 `git commit` 流程就构造一个 commit 对象并挂到某个 ref 上"——这正是 β 方案构造 vbranch 的方式（不 checkout、不 stage，直接写 tree + commit + ref）。

### Orphan Branch（孤儿分支）

普通分支的历史都能追溯到同一个初始 commit；**孤儿分支的首个 commit 没有父 commit**，和主历史完全隔离。

用 `git checkout --orphan <name>` 创建。经典用法：GitHub Pages 用 `gh-pages` 孤儿分支存网站产物，不和 main 历史混在一起。

Entire 和 β 方案用它存推理元数据：代码历史在 main，元数据在 `refs/agm/metadata/v1`（orphan），两者井水不犯河水，但通过 commit trailer 互链。

### Commit Trailer（提交尾注）

Commit message 末尾的结构化键值对，Git 社区的既有约定。你可能见过 `Signed-off-by: xxx`、`Co-Authored-By: xxx`——这些都是 trailer。

β 方案借用它做"代码 ↔ 推理"的双向链接：
```
feat: 加用户登录

实现了 OAuth 流程，参考 RFC 6749。

Entire-Checkpoint: a3b2c4d5
AGM-Agent: claude-code-session-42
```

`git log --format='%(trailers)'` 能专门提取 trailer。

### Integration Branch / Workspace Merge Commit（集成分支）

GitButler 的核心技巧：`.git/HEAD` 不指向某个普通分支，而指向一个**自动生成的 merge commit**，这个 merge commit 的父节点是所有激活的虚拟分支。

```
集成 commit (workspace)
  ├── parent: vbranch_1 HEAD
  ├── parent: vbranch_2 HEAD
  └── parent: vbranch_3 HEAD
```

工作目录显示的就是这个 merge commit 的 tree。任何 vbranch 变更 → 重算 merge → 更新集成 commit → 工作目录自动更新。

β 方案照搬这个做法，集成分支叫 `refs/heads/agm/workspace`。

### Virtual Branch（虚拟分支）

"虚拟"在于：**它不对应 `.git/refs/heads/` 下的任意一个普通分支**，而是通过 "集成分支 + 归属信息" 逻辑构造出来的分支视图。

- 物理上：存在 `refs/agm/vbranch/<id>` 里，是标准 Git commit
- 逻辑上：工作目录里所有"属于这个 vbranch 的 hunk"的集合
- Push 时：重构成标准 Git commit，远端看到的就是普通分支

所以"虚拟"只对本地来说是虚拟的，对远端透明。

### First-Class Conflicts（一等公民的冲突）

传统 Git：冲突是**错误状态**，`git merge` 会把你踢进 "MERGING" 状态，手上拿着带 `<<<<<<<` 标记的文件，必须解决才能继续。

GitButler 的 first-class conflicts：冲突是**数据**，可以：
- 跨 commit 保留、延后解决
- 在 UI 里标红但不阻塞继续工作
- 存到独立数据结构里，不污染文件内容

β 方案 MVP 阶段简化版：冲突检测到 → 存 SQLite → 暂停相关 agent，不往文件里塞 `<<<<<<<` 标记。

### Push 转换

虚拟分支在本地是 ref + 归属表，但远端的 GitHub/GitLab 只认标准 Git commit。Push 时要做转换：

```go
func push_virtual_branch(vbranch) {
    tree := base_tree.clone()
    for hunk in vbranch.ownership {
        tree.apply(hunk)
    }
    commit := git_commit(tree, vbranch.head, message)
    git_push(remote, vbranch.name, commit)
}
```

转换后：同事看到的是普通 PR，完全不知道你本地用了虚拟分支。

## 二、Go 技术栈

### go-git

纯 Go 实现的 Git 库，项目地址 `github.com/go-git/go-git`。

- 对标：C 语言的 libgit2 / Rust 的 `gitoxide`
- 优点：纯 Go，无 C 依赖，跨平台编译简单；plumbing API 丰富
- 缺点：某些 edge case 的成熟度不如 libgit2（特别是大仓库性能、稀疏 checkout、部分 pack 格式）
- β 方案用它做：打开仓库、读写 ref、构造 tree/commit/blob、做 merge

Entire 用 go-git 做 Checkpoints，是 β 方案选它的最大信心来源。

### fsnotify

Go 生态的跨平台文件系统事件监听库，`github.com/fsnotify/fsnotify`。

监听某个目录，当里面的文件被创建/修改/删除/重命名时，你的代码会收到事件：
```go
watcher, _ := fsnotify.NewWatcher()
watcher.Add("/path/to/repo")
for event := range watcher.Events {
    fmt.Println(event.Op, event.Name)  // e.g. WRITE /path/to/repo/src/main.go
}
```

底层在不同平台调用不同 API：
- Linux：`inotify`
- macOS：`FSEvents`
- Windows：`ReadDirectoryChangesW`（见下节）

β 方案用它捕捉"agent 改了哪个文件"。

### SQLite + WAL 模式

**SQLite**：单文件数据库，一个 `.db` 文件就是整个数据库，无需服务器。Git 仓库里放个 `.agm/state.db` 就够用。

**WAL（Write-Ahead Logging）**：SQLite 的两种写入模式之一。
- 默认（rollback journal）：写入时锁整个库，并发读写相互阻塞
- WAL 模式：写入先追加到独立的 `.wal` 文件，读不被写阻塞，读写并发性好

Windows 下文件锁比 Linux 严格，默认模式容易卡，**开 WAL 几乎是 Windows 上 SQLite 的标配**。开启方式：`PRAGMA journal_mode=WAL;`

### cobra

Go 写 CLI 的事实标准库，`github.com/spf13/cobra`。kubectl、hugo、docker CLI 都用它。

帮你自动生成：子命令（`agm init` / `agm start`）、参数解析、help 输出、bash/zsh 补全。

### bubbletea

Go 写 TUI（文本用户界面）的框架，`github.com/charmbracelet/bubbletea`。

采用 Elm 架构（Model-Update-View），适合做 lazygit、k9s 那类"看着像 UI 但跑在终端里"的程序。β 方案可选的 UI 形态之一。

## 三、文件系统与操作系统

### Copy-on-Write (COW) 文件系统

"写时复制"——复制一个文件时，只复制元数据（指针），不复制实际数据块；**只有当你真的去改动副本时，被改动的数据块才被复制**。

效果：复制一个 10 GB 文件，瞬间完成，磁盘占用几乎为零；改动几 KB，也只多占几 KB。

支持 CoW 的文件系统：
- macOS **APFS**（默认）
- Linux **Btrfs、ZFS**
- Windows **ReFS**（不是 NTFS，需要手动开启的服务器文件系统）

这也是为什么 γ 方案在 Windows 上行不通——**NTFS 不支持 CoW**，即使 Windows 11 也默认 NTFS。

### ReadDirectoryChangesW（Windows 文件监听 API）

Windows 原生的目录变化通知 API。`fsnotify` 在 Windows 下包装的就是它。

已知问题：
- 事件队列有限，**高频改动时会丢事件**（内核缓冲区满）
- 重命名有时只报一半（只有 RENAMED_NEW 没有 RENAMED_OLD）
- 符号链接支持诡异

β 方案的对策：**fsnotify + 5 秒轮询 + Claude hook checkpoint 三重保险**，任何一个漏了另外两个能补上。

### Symlink vs Hardlink（符号链接 vs 硬链接）

- **Symlink**（软链接）：像快捷方式，指向另一个文件路径。原文件没了，symlink 就失效。
- **Hardlink**（硬链接）：同一个文件的第二个名字。底层是同一块 inode，内容共享，删掉其中一个不影响另一个。

Windows 下创建 symlink **需要管理员权限**（或开启 Developer Mode），hardlink 不需要。β 方案避免 symlink，是为了让普通用户不踩这个坑。

### Long Path（Windows 长路径）

Windows 传统 API 有 `MAX_PATH = 260` 字符限制。路径超长（比如 node_modules 深层嵌套）时会操作失败。

解决方案：
1. 注册表 `HKLM\SYSTEM\CurrentControlSet\Control\FileSystem\LongPathsEnabled = 1`
2. 在可执行文件的 manifest 里声明 `<longPathAware>true</longPathAware>`
3. 用 `\\?\` 前缀的 UNC 路径绕过

β 方案在 Windows 上会遇到深层仓库，所以要显式处理。

### 文件锁（Windows vs Linux）

- Linux：advisory lock，进程可以选择尊重或无视
- Windows：**mandatory lock**，一个进程打开文件写，其它进程连读都可能失败

这就是为什么 SQLite 要开 WAL、git 在 Windows 上偶尔报 "index.lock exists"——都是 Windows 严格锁语义的后果。

## 四、Claude Code Hook 生态

Claude Code 在 session 生命周期的若干关键点会执行用户配置的 shell 命令，通过 stdin 传 JSON 给你，你的脚本处理完可以影响后续行为。这就是 **hook**。

β 方案的 hook 只是"被动监听 + 记录"，不改 Claude 行为。

### 四个核心 hook（β MVP 关注）

| Hook | 触发时机 | AGM 用途 |
|---|---|---|
| **SessionStart** | Claude session 启动 | 注册 agent、创建 vbranch ref |
| **UserPromptSubmit** | 用户按下回车发出 prompt | **做一次 worktree diff 快照**，作为归属判定的 checkpoint |
| **PostToolUse** | Claude 用完一个工具（读文件/写文件/跑命令） | 累加 token 消耗、检查预算熔断 |
| **Stop** | Session 结束 | 最终归属确认、生成 handover note、commit vbranch |

Claude Code 官方文档还列了其它 3 个（PreToolUse、Notification、SubagentStop 等），MVP 可以先忽略。

### Transcript（会话抄本）

Claude Code 把整个会话（每条消息、每次工具调用、每次工具返回）存成 JSON 文件，通常在 `~/.claude/projects/<project-id>/<session-id>.jsonl`。

每行一条 JSON 记录。hook 拿到的 session_id 可以据此定位 transcript 文件，读出"Claude 这一轮到底做了什么"。

Entire 说 transcript 是它的"单一事实源"，β 方案也会依赖它做事后归属确认。

### Token 预算与熔断

**Token**：LLM 读写的最小计费单位，大致一个英文 token ≈ 4 字符，中文 token 更短。一次 Claude Code 任务可能消耗几万到几十万 token。

**熔断**：从电路熔断器借来的词。达到阈值（比如 "本任务上限 50k token"）自动断开——在 AGM 语境下就是**暂停这个 agent、不让它继续花钱**。

β 方案的熔断点在 `PostToolUse` hook：每次工具调用后累加，超过用户设定的预算就停。

### Handover Note（交接笔记）

Agent 做到一半人类接手时的交接材料。内容至少包括：
- **当前状态**：已经改了哪些文件、完成到什么程度
- **下一步假设**：如果继续做，下一步应该是什么
- **卡点 / 疑问**：agent 没解决的问题

β 方案在 `Stop` hook 里让另一个 LLM 自动生成 handover note，存到 orphan metadata branch。人类接手时（或下一个 agent 接力时）读这份 note 开工。

## 五、架构与工具链术语

### CLI vs TUI vs GUI

- **CLI**（Command-Line Interface）：`git status`、`ls`——一次命令一次输出，无交互
- **TUI**（Text User Interface）：`vim`、`lazygit`、`htop`——终端里跑的全屏交互式界面，键盘导航
- **GUI**（Graphical User Interface）：普通桌面应用

β 方案 v0.1 走 **CLI + TUI 组合**：日常用 CLI（`agm status`、`agm start`），看板视图用 TUI（bubbletea）。

### Tauri

Rust 写的跨平台桌面应用框架，对标 Electron。

- 优点：打出来的应用体积比 Electron 小一个数量级（几 MB vs 几百 MB），启动快、内存占用低
- 缺点：前端仍是 WebView（Windows 用 WebView2、macOS 用 WKWebView），渲染不如原生
- GitButler 就是 Tauri + Svelte

β 方案 v0.1 不用 Tauri（单 binary 优先），v0.2 可考虑。

### FUSE（Filesystem in Userspace）

Linux/macOS 的机制：**让普通用户进程假装自己是一个文件系统**。Linux `sshfs`（把远端目录 mount 成本地目录）就是靠它。

在 β 方案里：δ 方案（内存工作树）想用它拦截 agent 的文件读写。Windows 没有原生 FUSE（有第三方 WinFsp 但依赖复杂）——这是 δ 方案在 Windows 上难落地的核心原因。

β 方案不用 FUSE，所以不受这个限制。

### libgit2

C 语言的 Git 实现库，GitHub 官方维护。GitButler 基于它。

和 go-git 是"同类产品，不同语言生态"的关系。β 方案选 go-git 是为了：
1. Go 单 binary 分发比 C 链接动态库简单
2. 对齐 Entire 的技术栈
3. 避免 cgo 在 Windows 上的额外复杂度

### MVCC（多版本并发控制）

数据库领域的概念：**同一时刻允许多个事务看到不同版本的数据**，避免读写互相阻塞。PostgreSQL、SQLite、MySQL InnoDB 都用它。

在本项目语境：δ 方案"内存工作树"的思路借鉴了 MVCC——每个 agent 看到自己的"版本"，底层通过版本号调度。β 方案**没用 MVCC**，这里只是帮你看懂之前推演里的类比。

### OverlayFS

Linux 的分层文件系统：把多个目录"叠"在一起呈现成一个目录，上层的修改不影响下层。Docker 镜像层、Android 的 A/B 分区都用它。

同样是δ 方案的类比，β 方案不用。

## 最后一句话

如果你只要记住三件事：

1. **Hunk = 一段代码改动**。归属到谁决定了这段改动算谁的虚拟分支。
2. **Ref = Git 里的指针**。β 方案在 `refs/agm/*` 下玩自己的命名空间，不影响 `refs/heads/*` 的用户分支。
3. **Hook = Claude Code 生命周期的 4 个钩子**（`SessionStart` / `UserPromptSubmit` / `PostToolUse` / `Stop`）。AGM 的所有自动化都挂在这四个点上。

其它都是为了支撑这三件事存在的工具。
